Introduction

Hitec telemetry was build because Hitec only delivers a solution for iPhone users.

The existing applictaion is connected with a special cable based adaptor. This solution used wireless bluetooth connection.

The disadvantage is the need of an extra build module based on arduino, which communicates with the Aurora 9 and the bluetooth module.

See section Building your own bluetooth module. The costs are moderate and the expected skills for building are on a low hobby level.

There is an alternative for slope flyers and soarers, a module, that connects directly to the Hitec data port and schows minimal values like voltage, current power and power consumption. Don't hestitate to contact via feedback form for further datails.it is based on Arduino board using I2C communication and SPI interface for graphics driving a SSD1306 OLED monochrom display from Adafruit.

First steps

The following steps will show an initial setup scenario. It is recommended, that you have successfully build your bluetooth module as mentioned here:Building your own bluetooth module.

Power on your radio and the module. Pair your module using password '0000' or '1234' in case you did not changed it in the configuration of your module.

Start the Hitec telemetry application on your smartphone. The application shows some messages regarding the database version and the choosen profile.

For the first step that is profile 'default'.

Since version 1.07 we do not connect automatically anymore. Chosse connect (bluetooth icon) from the action bar. May be overwritten by setup option. see advanced functions.

A status message will inform you about the actual connecting condition. Normally it schould read 'connected to HITECAURORA' or whatelse you have named your module.

In any other case check your connection and setup of the module. (Power, Pairing and so on.)

You should immediately see the voltage value turning from value 0 to the actual value transmitted through the Hitec telemetry system.

Proceed with the advanced configuration described in the next scetion to benefit from all avaiable configurable options.

Advanced Details settings and options

- onTouch events on main screen

  1. voltage: short touch toggles between live,minimum and maximum values, long touch resets all values and toogles sound on flag
  2. Amps: short touch toggles between live,minimum and maximum values, long touch resets all values
  3. Power: short touch toggles between live,minimum and maximum values, long touch resets all values
  4. Height: short touch toggles between live,minimum and maximum values, long touch resets all values from height and Climbrate
  5. ClimbR: long touch resets all values from height and Climbrate

- options

  1. threshold
    You can setup the threshold for every configurable sensor value. The logic is predefined and will be user editable in a future version. E.g. voltage will always be below the stored value, all other will be handled  as above.
     
  2. general
    1. alarms on: flag to enable alarms
    2. speak alarms on: flag to enable speaking alarms, must be configured under speech setup
    3. sound file: sound picker for alarm tone, default is deliered with app
    4. sound on: flag for sound, in case alarm occurs, will not overwrite alarms on flag
    5. bluetooth device: name for bluetooth device default is HITECAURORA
    6. log file name: name for log_file default is spectra_view.txt
    7. logging enabled: flag for enbling raw data logging
    8. speech on: flag for speech sensor value periodically, needs other setup options
    9. speech intervall ms: intervall for speaking sensorvalues 30000 means every 30s
    10. speech alert intervall ms: intervall for speaking sensor alarms 15000 means every 15s
    11. speech volume %: volume setup up for speech system in %
    12. vibrate on: additional signal for alarms does not overwrite sound on
    13. use metric units: toogle between metric and non metric units for altitude, speed, temperature, climbrate, will be used in normal nd vario screen, not in vario test
    14. auto connect: flag for autoconnet to bluetooth device during startup.
       
  3. single_alarms: Check every sensor value to be included in alarm monitoring
  4. speak_alarms: Check every sensor value to be included alarm monitoring and periodically  speaking
  5. units: Setup special units which override defaults for rpm1,2 and temp1,2. may be expanded to other values sometime
  6. lables: Setup special lables which override defaults for rpm1,2 and temp1,2. may be expanded to other values sometime
  7. hysteresis: Some values needs hysteresis, due to model specifications and conditions to avoid false alarms. Values of 50 are good practice, matters that the value has to be 50 times above or beneath the given threshold.
  8. sensor_settings: Specify the applied type of current sensor, C200, C50 for Hitec sensors Unilog / Unisens for sensors from SM-Modelltechnik. You can use them with a special interface directly with the telemetry port of any Optima 7 or 9.  DIY current senors ahve to be setup as C200.   This is a diy current sensor based on a arduino board and a ACS709 hall sensor.
  9. speech setup: setup values to be spoken periodiacally, setup up text for alarms any periodic text, must be adjusted to your needs and to the technical prerequisites of your device.

- test speech

  • test speaking values, if there is something to do - needs proper setup

  • enable raw logging, overwrites option value

  • vario function, still under construction, needs some more reverse engineering of the Hitec protocol and calculations of the delivered pressure values

- data explorer

- graphs

  • show some graphs with logged data (live, or choose new datafile from dataexplorer)

- vario test

  • test vario to be familar with its special functions. Has slider to immitate the altitude.

- backup database

  • backup database to sdcard in sqlite format

- version information

  • show version and development terms

M-Link users

Software version 1.09 offers support for M-Link radios such as EVO PRO and SX.

Check 'MPX protocol' under general settings and restart app.

The M-Link protocol is simple but therefore very fexible. Hitec has another concept and does not use sensorclasses and alarm settiings in their datagram.

So for more simplicity there is only a sensor mapping from M-Link to Hitec and there are three new values: LQI (line quality), txstate (trasmitter state such as rangecheck lost connection and avaiable downlink) and firmware (softwareversion of tx module).

M-Link offers more features which will be implemented in further versions (events on switches of the radio e.g.).

The interface uses only the transmitter module, there is no need for an arduino board. Update the tx module to softwareversion 0.44.

Some older modules have an resistor of 100 Ohm in the VCC line. Unsolder it and solder in a bridge instead or use silver paint to short the contacts.

The curent is limited to about 33 mA (3,3 V/100 Ohm) many bluetooth module may work fine, but some need modifying the board as mentioned above.

Connect GND,VCC and Signal to the Bluetooth module using a JR servoplug. Connetc to GND,VCC (3.3V) and RX. Do software setup in the same way like module for Hitec (115200 baud).

That's all for the moment.

The find model option uses for MLink the course (referenced to north: east complies to 90 degrees, west to 270 degrees) and the distance from the start point as reference.

The coordinates for googlemaps are based (and calculated) on standard trigonometrical considerations.

Please let the azimuth angle shown as course on the display to calm down a bit, since to technical limitations it oscillates more or less a bit.

One minute should be ok for that. It is presumed that you stay at the starting point of your model. Then start the maps activity. The estimation for the current position needs 4 or 5 transits, then it can be assumed to be stable. After that the model position is calculated and a marker will appear.

Move to your model, the actual position will be updated on every change under the title.

Don't expect too much accuracy from this tool. The calculation delivers approximate values.

The calculation for the 'Grosskreisbogen' to determine the models position uses the following:


private void MPXLocation(double lat1,double lng1, double alpha, double distance){
double lat2= 0.0;
double lng2 = 0.0;
lat1= Math.toRadians(lat1);
lng1= Math.toRadians(lng1);
double c = distance/6371000;
double a = Math.acos(Math.sin(lat1) * Math.cos(c) +Math.cos(lat1)*Math.sin(c)*Math.cos(Math.toRadians(alpha)));
double gamma = Math.acos(((Math.cos(c) -Math.cos(a)*Math.sin(lat1)) / Math.sin(a)/Math.cos(lat1)));
lat2 = Math.PI / 2;
lat2 = lat2 - a;
if (Math.toRadians(alpha) < Math.PI){
lng2 = lng1 + gamma;
} else {
lng2 = lng1 - gamma;
}
Latitude = Math.toDegrees(lat2);
Longitude = Math.toDegrees(lng2);
SetLastLocation(Latitude, Longitude); // function to addmarker
}

Building your own bluetooth module

First you should get a blueooth module / shield which capable of passing serialdata to bluetooth.

I used this one avaiable from reichelt electoronic in germany.

Many other suppliers may have an identical module in stock. NIBO BEE BLUE :: Bluetooth Modul für NIBObee.

The advantage of this module is the integrated voltage regulator, so you can supply any voltage from 3,5 to 9V. e.g. using a 2s Lipo or the build in battery from the Aurora 9.

Second buy an Arduino board, use Arduino Mini Pro, which is avaible e.g. from exp-tech in germany : Arduino Pro Mini 328 - 3.3V/8MHz

Please check out that you use the 3.3V version, since Hitec data port uses 3.3V logic level!

The bluetooth module has to set up in communication, the transfer speed (baudrate) must be set to 115.200 Baud.

The name should be changed to 'HITECAURORA'.

This is default in the application. On Connect, only HITECAURORA named modules are listed.

Nevertheless you can change the connectionstring in the preferences of the app.

In most cases you can configure the module with traditional AT commands (see documenttion of your module).

Connect the module to your PC with a serial to usb bridge and initiate a connection with a proram like hyperterminal an windows machines.

The standard parameters are 9600/8N1, ID: linvor, Password: 1234.

Enter 'AT' Return, you should get OK as answer. Now change Baudrate to 1115.200 with AT+BAUD8.

Change Name to 'HITECAURORA' with AT+NAMEHITECAURORA. Change Pin to whatever you like with AT+PIN0000 (leave it for default '1234').

You will need the PIN when pairing the device with your smartphone. Solder 1k Resitors (use SMD type if avaiable) to SCL und SDA Port (A4,A5 printed on Arduino board) to VCC (this is 3.3V don't use RAW!)

Take a standard servo connection and solder the orange cable to A5, the red cable to A4 and the black cable anywehre to ground.

Connect the TX pin from Arduino to the RX pin of the bluetooth module and RX pin from Arduino to TX pin of the bluetooth module.

The second is optional, we only send data and never handle received bytes. Connect power supply lines from RAW or your battery to the bluetooth module. (GND and VCC).

Program the Arduino board with the latest firmware. (V1.1). The sourcecode is here:

#include avr/io.h
#include TWI_Slave.h
uint8_t buffer[128];
uint8_t bytesIn = 0;
int j = 0;
boolean bufferready = false;
int volt = 0;
boolean flip = false;
int waitCount = 0;
void setup()
{
pinMode(13, OUTPUT);
Serial.begin(115200);
TWI_AttachSlaveRxEvent(readI2c);
TWI_Slave_Initialise( 8, true);
TWI_Start_Transceiver();
sei();
}
void readI2c(uint8_t* inBytes, int numBytes)
{
for(uint8_t i = 0; i < numBytes; ++i)
buffer[i] = inBytes[i];
bytesIn = numBytes;
bufferready = true;
digitalWrite(13, (flip?HIGH:LOW));
flip = !flip;
}
void loop()
{
while (true)
{
if (bufferready)
{
bufferready = false;
for (int x = 0; x < bytesIn; x++)
{

Serial.write(buffer[x]);
}
}
else
{
delay(10);
}
}
}

Now connect the power supply, the blue LED should blink.

Insert data cable into the data port of the Aurora 9 (backside on the Spectrum module).

Power on the Aurora 9. The data will be send only if a telemetry receiver is connected and powered.

An Optima 6 receiver will be enough for the first steps.

If the connection is established, the Aurora beeps twice and the green LED on the Arduino boards begins to flash fo every valid datapacket received.

That's all for the moment.

To use the OpenFormat (short name: OF) you have to create an INI file and an optional picture for your device. The INI file contains all the needed configuration information for LogView to display your serial data. The format OF is pure ASCII data where the items are separated by ";". An OF packet starts with "$", then has several values separated by ";" and always ends up with CR + LF (#13#10). An example of a packet your device may produce is:

$1;1;0;1133;318;0;0;0;0;0;20;21;0;16
